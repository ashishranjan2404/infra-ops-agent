# 10 — Feedback for the next task

When a task depends on sibling task outputs (G5/G6/G7 here), check them first — in a parallel
Ralph run they are often still empty, so plan to fall back to the underlying verified repo
artifacts and make every claim auditable (a small `proof_points.json` mapping claim → value →
source path is cheap and turns a "pitch" into something a reviewer can check). For any
investor/partner or "why we're different" writing task in this repo, the honest differentiator
is the verifiable environment + the trap-aware/self-learning verifier, NOT the REx refinement
loop — the repo's own `docs/headline_insights.md` ablation shows REx's raw lift was mostly
oracle-feedback leakage, so leading with REx numbers will get torched; lead with the env/verifier
and frame REx by its escalation/safety behavior. Keep persuasion copy under ~700 words, source
every number inline, and put the unflattering ablation ON the page — in this codebase, honesty is
a genuine selling point against demo-ware.
