## 1. Motivation

**For what purpose was the dataset created?**
SRE-Degrees is a benchmark of synthetic, simulator-backed Site Reliability Engineering
(SRE) incidents. Each scenario injects a single *hidden root fault* into a small service
topology that then *cascades*, raising alerts on downstream nodes that are **not** the
cause. The task an agent must solve is **root-cause localization under cascading alerts**:
state the true mechanism, avoid red-herring explanations, apply the canonical fix, and
avoid "trap" actions (e.g. scaling the symptomatic service instead of fixing the root).

The central construct — *"the loudest alert is not the cause; the smoking gun is buried"* —
is encoded as explicit per-scenario assertions and is satisfied by **40 / 51** scenarios in
the current snapshot. It exists to train and evaluate (frozen-LLM) agents and REx tree
policies on diagnosis quality, with a **deterministic, reproducible** reward signal.

**Who created it / funded it?** The SRE-Degrees / REx research project (this repository).
Scenarios are generated by the in-repo **CIDG** (Cascading-Incident Data Generator)
pipeline. No external funding metadata is recorded in the corpus.

## 2. Composition

**What do instances represent?** Each instance is one *incident scenario*, a single YAML
file under `scenarios/cidg/generated/`. A scenario fully specifies a service topology,
one hidden root fault, the SLOs that breach, the buried smoking-gun signal, distractor
"trap" actions, and the canonical fix — enough for a deterministic simulator (`sim/engine.py`)
to replay it and for the REx scorer (`rex/scoring.py`) to grade an agent's diagnosis.

**How many instances?** **51** scenario YAMLs on disk; **32** are indexed in
`registry.json` (see §8 — Registry gap). All numbers in this section are from the snapshot.

Classification heuristic: a scenario is `postmortem_derived` iff `meta.source`/filename

`-positive`, `singleton`). This is a documented **heuristic**, not a hand-verified label —
audit with `compute_stats.py`.

**Platform / fault encoding.** Every scenario declares `clouds: [gke, lke]` (51/51) and
`fault.chaos_kind: exec` (51/51). The benchmark is therefore **single-platform-shaped**
(Kubernetes on GKE/LKE) and **single fault-injection mechanism** (`exec`).

### 2.6 Registry families (32 indexed scenarios)

`simple` 8 · `cascade` 14 · `novel` 10. The `novel` family holds the named historical
incidents intended to test generalization beyond the simple/cascade training templates.

**Labels / ground truth.** Each indexed scenario carries `gold_root` (the correct
mechanism, in natural language), `red_herrings[]` (distractor explanations to reject),
`fix_tools[]`, `traps[]`, `forbidden[]`, and a `symptom`. The non-indexed scenarios carry
the same ground truth inside the YAML (`root_cause`, `canonical_fix`, `observation.smoking_guns`).

**Missing information / noise.** `meta.urls` is **empty for every scenario** — postmortem
provenance is recorded only as a free-text title in `meta.source`, not as a citable URL.

**Splits.** No official train/test split file ships with the corpus. Recommended practice:
treat `synthetic_leaf` + `simple`/`cascade` families as in-distribution training, and the
`novel` postmortem-derived scenarios as held-out generalization — and additionally report
the `synthetic_leaf` subset separately as a contamination-controlled measurement (§Appendix C).

## 3. Collection Process

**How was the data collected?** Scenarios are **not** logged real telemetry. Each is a
*hand-curated abstraction* of a public incident (for `postmortem_derived`) or a generic
fault template (for `synthetic_leaf`), encoded into the CIDG schema and rendered runnable
by the simulator. The pipeline:

1. **Source selection.** A public postmortem (or a generic mechanism) is chosen and its
   failure mode mapped to one of ~15 `failure_class` values.
2. **Topology abstraction.** The real architecture is reduced to a 1–5 node graph
   (`control_plane` / `lb` / `service` kinds) with typed edges (`required` / `discovery`).
3. **Fault injection.** A single hidden `root_cause` is placed on one node; `fault` encodes
   an `exec` chaos action that flips a simulator state flag for `duration_s` seconds.
4. **Cascade + observability.** `slo` breaches are declared on downstream nodes; one
   `smoking_guns` entry marks the true signal and `buried_under` quantifies how many louder
   distractor signals hide it.
5. **Traps + gold fix.** `trap_actions` (plausible-but-wrong remediations) and a
   `canonical_fix` (the correct tool + ordering) are attached.
6. **Determinism.** A per-scenario integer `seed` plus `chance` params (`flap_prob`,
   `jitter`, `partial_recovery_prob`) make simulator replay reproducible.

**Construct-validity caveat.** Solving a CIDG scenario means localizing a root fault in a
*simulated, abstracted* topology — a **necessary-not-sufficient proxy** for real incident
response. The mapping from real postmortem to sim is lossy by design.

**Over what timeframe?** The referenced real incidents span ~2013–2024 (e.g. Knight Capital
2012, Cloudbleed 2017, Facebook BGP 2021-10-04). The corpus itself is generated and
versioned in-repo; the snapshot date is stamped in `stats.json`.

## 4. Preprocessing / Labeling

No tokenization or featurization is applied — scenarios are consumed as structured YAML by
the simulator and scorer. The *labels* are authored as part of generation:
- `gold_root` / `root_cause.kind` — the correct mechanism.
- `red_herrings[]` — distractor causes the judge must reject.
- `observation.smoking_guns[].buried_under` — difficulty knob (how many louder signals
  hide the true one).
- `trap_actions[]` / `forbidden[]` — actions that incur the reward penalty.
- `canonical_fix.steps[]` + `ordering_notes` — the gold remediation.

Raw YAML is retained and is the canonical artifact; `registry.json` is a derived index over
a subset.

## 5. Uses

**Intended uses.**
- **RL / policy training & evaluation** of incident-diagnosis agents (REx tree search,
  frozen-LLM agents) with a deterministic reward.
- **Eval harnesses:** `rex/eval_pass_at_k.py`, `rex/harness_synth.py`.
- **Ablations** over difficulty (`buried_under`), topology size, and provenance class.

### 5.1 How items are scored (the reward contract — `rex/scoring.py`)

```
reward = 0.30·diagnosis_correct + 0.25·correct_fix_present + 0.45·resolved
         − 0.60·trap_or_forbidden_action_hit
```
- **Gold label per item** = `gold_root` (correct mechanism) vs `red_herrings[]` (distractors).
- **Default judge = `deterministic_judge`**: a discriminative keyword-stem classifier.
  An answer is `CORRECT` iff it hits **more gold-mechanism stems than red-herring stems**
  (and ≥ 1 gold stem). Pure, hermetic, no network — **reproducible across labs**.
- **`mechanism_score`** gives a continuous `[0,1]` version (gold fraction of discriminative
  hits) for finer within-group reward spread.
- **Judge mode** is `REX_JUDGE_MODE ∈ {deterministic (default), hybrid, llm}`. **`hybrid`
  and `llm` call an LLM judge on borderline cases and are therefore NON-deterministic** —
  do **not** use them for reproducible headline numbers. Report the judge mode with any result.

**Threats to scoring validity (read before trusting a number):**
- The deterministic judge is a *keyword* matcher. A correct **paraphrase** that misses the
  gold stems can be scored wrong; an answer that **keyword-hacks** gold stems without
  mechanistic understanding can be scored right. Mitigated (not eliminated) by stemming +
  synonym folding + dropping non-discriminative shared tokens.

**Out-of-scope / discouraged uses.** Do not use as a measure of *real* production incident
competence without external validation. Do not report `llm`/`hybrid`-judge numbers as
"reproducible."

## 8. Reproducibility Appendix (for AAAI)

### A. Regenerate every number in this card
```bash

        --dir scenarios/cidg/generated --out stats.json
```
Exits non-zero if any YAML or `registry.json` fails to parse. Output includes

### B. Scenario schema (every file conforms)
`meta{id,title,source,urls,failure_class,clouds}` · `topology{nodes[],edges[]}` ·
`root_cause{location,kind,severity,hidden,persistent,reset_by}` ·
`fault{chaos_kind,exec_cmd,sim,duration_s}` ·
`observation{alerting,monitoring_degrades,smoking_guns[]}` · `slo[]` · `trap_actions[]` ·
`canonical_fix{ordering_notes,steps[]}` ·
`assertions{cascades,loudest_alert_not_cause,fix_resolves,buried_gun_exists,hysteresis,monitoring_degrades}` ·
`chance{flap_prob,jitter,partial_recovery_prob}` · `seed:int`.

### C. Recommended reporting protocol (contamination control)
1. Report overall score **and** the `synthetic_leaf` subset score separately. The 31
   `postmortem_derived` scenarios name real incidents (Facebook BGP, Cloudbleed, Knight
   Capital, …) that frontier LLMs have very likely **read** — high scores there may reflect
   **memorization**, not diagnosis. The 20 `synthetic_leaf` scenarios carry no real name and
   are contamination-safe.
2. Pin and report `REX_JUDGE_MODE` (use `deterministic` for reproducible numbers).
3. Pin the corpus by git commit + `snapshot_utc`; report `n_yaml` and `n_registry`.

### D. Determinism
Per-scenario `seed` + `chance` params make the simulator replay deterministic. The
deterministic judge adds no sampling noise. Therefore, for a fixed agent, seed, judge=
`deterministic`, and corpus commit, scores are reproducible bit-for-bit.

### E. Known gaps / limitations (honest)
1. **Synthetic, not telemetry.** Scenarios are abstractions; external validity is a
   deliberate trade-off for reproducibility (no real logs ⇒ labs can reproduce numbers).
2. **Single platform / single fault mechanism.** 51/51 are GKE/LKE Kubernetes with
   `chaos_kind: exec`. No bare-metal, serverless, multi-region, or non-`exec` faults.
3. **Small topologies** (1–5 nodes) bound difficulty vs. production fleets.
4. **Contamination risk** on the 31 named historical incidents (see §C).
5. **Keyword judge fragility** — paraphrase miss / keyword-hack (see §5.1).
6. **Empty `meta.urls`** — provenance is a free-text title, not a citable link; not
   independently verifiable from the corpus alone.
7. **Registry lag (the registry gap).** `registry.json` indexes only 32 of the 51 YAMLs;
   19 scenarios (snapshot-dependent; see `stats.json.yaml_not_in_registry`) are present on
   disk but **not in the registry**, so registry-driven eval harnesses silently skip them.
   Regenerate the registry before reporting coverage.
8. **Unused schema fields** — `hysteresis` and `monitoring_degrades` are declared but set on
   0 scenarios; those difficulty dimensions are untested.
