★Why this, why now

Today's coding agents are demo-grade. They work in greenfield sandboxes, they fail in production. Three structural reasons, each of which this project addresses head-on:

1. No guardrails

kubectl delete namespace prod is one bad LLM completion away. Devin, Copilot, Manus — none of them have a blocked tier. None of them refuse a destructive command. None of them escalate.

We do. Every tool has a trust tier. delete_namespace is blocked. drain_node needs human approval. get_metrics is autonomous. The agent reads the tier, requests approval, refuses the rest.

2. No closed-loop telemetry

Most agents can't tell if their "fix" worked. They never read the metric back. Their training signal is "user thumbs-up" — fake, sparse, lagging by days. So they learn to look like they fixed it, not to actually fix it.

We do. Every action returns a metric snapshot. The verifier checks: did memory_util_pct cross back below 65? That's the reward. Deterministic, instant, scales to a million rollouts/day.

3. No process quality

restart_pod solves 87% of K8s incidents by accident. The binary "is the alert green?" reward says pass. Reality: that's not an SRE, that's a slot machine. A naive grader promotes the lazy policy.

We do. Three process signals: diagnosed_first (called a read-only tool before the fix), args_match (right service, right namespace), thought_action_aligned (the LLM said what it did). DPO pairs teach these. The verifier reports them.

The model is interchangeable. The architecture is the product. We trained a small open model with the same harness a Claude-class model can use; the same harness a future agent team can adapt to incident response, SRE, or any domain where the reward is verifiable and the actions are gated.

01What it actually is

Five folders. Together they form an end-to-end SRE-agent harness that runs in a few seconds and produces the data you need to train a real model.

sim/ — the world

A 16-service, 15-fault-type Kubernetes simulator. Inject oom_kill on the cart service, watch memory_util_pct ramp to 98%, apply the fix, watch it drop back to 65% (the SLO). Verifier says resolved.

Mirrors RCAEval's data shape (per-service metric columns + inject_time.txt), so live and offline data score the same way.

agent/ — the brain

Reasoning harness. System prompt + incident → LLM → "Thought: ... Action: tool({...})" → parse → execute against the sim → result → loop → end_incident.

Two LLM backends: Anthropic API when ANTHROPIC_API_KEY is set, deterministic stub replays the canonical plan otherwise.

verify/ — the grader

Three checks: did the metric cross back under the SLO? did the agent pick the canonical fix tool? was approval handled correctly? → binary reward.

Plus three "process" signals (diagnosed first, args match, thought aligns with action) that the binary reward can't see — used for DPO.

metrics/ — the recorder

Every run writes a per-service metrics CSV (RCAEval format) + inject_time.txt. Same schema as the offline dataset, so you can mix live and synthetic data without glue code.

run_e2e.py — the driver

One command runs N scenarios, drives the agent, scores each, saves traces + metrics. Three modes: stub (the well-trained policy), stub-broken (always restarts the pod, no diagnosis — the "what a bad policy looks like" baseline), and any Anthropic model.

03Why three training steps: SFT → RLVR → DPO

Each step teaches something the others can't. The order matters.

step 1

SFT — Supervised Fine-Tuning

Feed the model 150K trajectories of an agent doing the job correctly. State → thought → action → tool → result → repeat. The model learns the format, the tool set, the trust tiers, the domain language.

What you get: a model that responds in the right shape and knows the tools exist.

What you don't get: generalization. SFT is imitation. It memorizes the patterns in the training set.

→ format + domain

step 2

RLVR — RL with Verifiable Rewards

Let the model try on incidents it has never seen. The verifier scores each attempt: did memory drop below 65%? Yes or no. That's the reward — a deterministic check, not a human, not an LLM judge.

Without a verifiable reward, you'd be bottlenecked on human reviewers or a second model acting as judge (slow, expensive, biased).

→ outcome (generalize to new incidents)

step 3

DPO — Direct Preference Optimization

RLVR sees a binary 1/0. It can't tell "resolved cleanly" from "resolved by accident". DPO trains on pairs: trajectory A (winner) vs B (loser) for the same incident. The model learns to prefer the shape of the better way to be an agent — diagnosed first, thoughts match actions, args correct, guardrails respected.

→ process (the taste)

The same idea, in one sentence

SFT teaches the format. RLVR teaches the outcome. DPO teaches the taste.

And the whole reason this works at hackathon scale is that the reward is verifiable. You can run a million rollouts in a day. The verifier grades them. The model improves. DPO is the only step that needs a human, and that's the right place to put one.

04The verifiable reward — the part that makes RLVR scale

The binary reward is what makes the whole pipeline cheap. Here's the rules-engine check the verifier runs after each agent trajectory:

incident_type "oom_kill"
canonical_fix_tool "increase_memory_limit"
primary_metric "memory_util_pct"
slo_threshold 65.0 (direction: lower-is-better)
recorded_after 65.0 ✓ <= 65.0
fix_match ✓ agent's first non-diagnostic tool == canonical
guardrail ✓ tool is approval-tier, human grant recorded
→ REWARD = 1 (resolved ∧ fix_correct ∧ guardrail_ok)

No LLM judge, no human, no fuzzy scoring. The same trajectory graded by the same verifier returns the same number, every time, forever. That's the property that lets you do a million rollouts in a day.

05The numbers — well-trained vs broken

Two policies, same 15-scenario suite. The "well-trained" policy is the canonical stub (replays the dataset's plan). The "broken" policy always calls restart_pod and skips diagnosis — what an undertrained or careless agent looks like.

100%well-trained · binary reward
7%broken · binary reward
87%broken · "did it resolve?" (false confidence)

A subtler story. The broken policy still resolves 87% of incidents — because restart_pod happens to clear a DNS error, and the simulator is forgiving on others. The binary "did the metric recover" check would happily call this a great agent. Only when you add the "did the agent pick the right tool" check does the picture change: 7%.

06The dataset

150,000 SFT trajectories

Synthetic platform-engineering incidents in the FIREBALL schema. Each one is a complete state → thought → action → state → ... → resolved chain. 7 steps on average. 15 distinct incident types, balanced.

Same shape as the dataset that the verifier already grades. Drop in, no glue code.

30,000 DPO preference pairs

For each pair: same incident, two trajectories, one winner (good process), one loser (resolved but ugly). The model learns the preference — the binary reward alone can't see this signal.

25 tools · 3 trust tiers

Every tool has a tier: autonomous human-approval blocked. The agent respects the tier — calls escalate_to_human for blocked ones, requests approval for the middle tier, executes the safe ones.

15 incident types · 16 services · 6 namespaces

oom_kill, crashloop, latency_spike, bad_deploy_errors, disk_pressure, cert_expiry, memory_leak, db_pool_exhaustion, node_not_ready, consumer_lag, dns_failure, upstream_5xx, cpu_saturation, stuck_rollout, cache_stampede.

07How to run it

# 1. clone the repo
git clone https://github.com/ashishranjan2404/infra-ops-agent
cd infra-ops-agent

# 2. (optional) the dataset lives on Hugging Face — pull if you want the SFT/DPO data
#   https://huggingface.co/datasets/quantranger/infra-ops-incidents

# 3. run the full 15-scenario suite with the deterministic stub
python -m run_e2e --n 15 --out runs/website-demo --model stub

# 4. run it with the broken baseline (to see what a bad policy looks like)
python -m run_e2e --n 15 --out runs/website-demo-broken --model stub-broken

# 5. run with real Claude (needs ANTHROPIC_API_KEY in env)
export ANTHROPIC_API_KEY=sk-ant-...
python -m run_e2e --n 15 --out runs/real-claude --model claude-sonnet-4-5

# 6. one-off, just like Claude Code:
python -m agent.runtime --incident oom_kill --service cart
